# FireRed Vision Encoder — Training Design

## Goal

Train a **FireRed-specific vision encoder** for a game-playing agent. It is *not* a general VLM — we
*want* it to memorise the FireRed world (map layouts, building locations, geometry). The encoder turns a
single overworld frame into a structured scene description the agent can plan over, with the primary
outputs being per-tile walkability / terrain grid, a prediction of where we are on the map (map loc, 
coordinates), predictions of all sprites and their location, etc.

We deliberately bake map knowledge into the weights. Static lookup trivia (path-finding, encounter
tables, type distributions) stays *out* of the weights and is served by retrieval against
`oggen3_data/` keyed on the predicted location — see [Derived layer](#derived-layer).

We aim to predict the json and map as free text, rather than a simple classification/segmentation head on 
its own, because the objective of the VLM is to render visual information in a legible format to the language model. 
The purpose of the vision encoder is to represent the game state to an LLM with the minimal possible tokens; 
we learn that by ensuring the LLM can recapitulate our structured information as free text.

This, clearly, would disturb the LLM if it was trainable. I am concerned that complete end-to-end training of the
LLM and vision encoder would lead to degradation of the base LLM, vision model would be learning towards,
leading to entanglement between two suboptimal models.. I want my VLM to have general utility;
it should be able to bolt-on to a large model and small model equally well, providing a small adaptor that translates
its representation into different spaces

To get around this, I only train LoRA-adapted LLMs. This gives flexibility in the LLM's output without disturbing
the base model. Obviously, we train the vison encoder end-to-end.

---

## Core principle: RAM is the label factory

We do **not** hand-caption screenshots. For every frame we dump the emulator's memory at the same
instant and serialise it to JSON — that JSON *is* the ground-truth label, that we predict via autoregression. 
Every target below has a direct RAM/ROM source (decomp names given for reference). 
Labelling is therefore free, dense, and pixel-perfect.

Pipeline: `automated run → (frame, RAM dump) pairs → serialise RAM to label JSON → train`.

---

## Architecture

One shared backbone, multiple heads (equivalently: one fine-tuned VLM emitting a constrained JSON
schema, one segmentation head). Build the **walkability grid head first** — it is the most learnable and the most useful to
a planner.

- **Segmentation head** → the tile grid (primary, below).
- **Categorical heads** → location id, map context (outdoor/indoor/cave).
- **Object overlay** → read from the live object-event table at runtime, not generated by the encoder.
these are sprites, pokeballs, destructibles/movables.
- **warp points**

Split every output into `perceived` (the encoder predicts it from pixels) vs `derived` (computed from
`perceived.map_id` via external tables). Never put derived trivia into the generation/segmentation
target.

## Primary target: the tile / walkability grid

The screen is 240×160 px = **15×10 metatiles** (16 px each). The target is a player-centred **15×10
class grid** — a semantic segmentation map. Source: metatile **collision** bit + **behavior** byte
(`MapGridGetCollisionAt(x,y)`, `MapGridGetMetatileBehaviorAt(x,y)`).

### Static terrain classes (from collision + metatile behavior)

| Class | Meaning | RAM / behavior source |
|-------|---------|------------------------|
| `free` | Walkable, no effect | collision = 0 |
| `blocked` | Impassable static geometry (buildings, fences, trees, walls, rocks) | collision ≠ 0 |
| `tall_grass` | Walkable **but triggers wild encounters** (nuzlocke-critical) | `MB_TALL_GRASS` |
| `water` | Blocked unless Surf | `MB_*WATER*` / surfable behaviors |
| `ledge_up_down` | One-way jump **downward** (south) | `MB_JUMP_SOUTH` |
| `ledge_down_up` | One-way jump **upward** (north) | `MB_JUMP_NORTH` |
| `ledge_left_right` | One-way jump **rightward** (east) | `MB_JUMP_EAST` |
| `ledge_right_left` | One-way jump **leftward** (west) | `MB_JUMP_WEST` |
| `warp` | Door / stairs / cave entrance — transitions to another map | warp-event table for the map |

Ledges are their own four classes because passability is **directional** — collapsing them into
`blocked` or `free` loses the only information that makes them useful for routing.

### Warp points

`warp` cells come from the map's **warp-event table**, where each entry is `(x, y) → destination map`.
This also gives **building identity for free**: the door's destination map (e.g. `POKEMON_CENTER_1F`,
`*_MART`, `*_GYM`) tells us what the building is *without entering it*. Expose this as an optional
attribute on `warp` cells:

```json
{ "tile": [x, y], "class": "warp", "destination": "POKEMON_CENTER_1F" }
```

### Destructibles / movables overlay (field-move obstacles)

These are **object events** (sprites with a graphics id), not metatile behaviors — read them from the
live object-event table, keyed on `graphicsId`. They block movement and disappear once used, so they
are part of the **runtime object overlay**, not the static terrain grid. Mark their tile in the grid
with the matching class:

| Class | Field move | Object source |
|-------|-----------|---------------|
| `cut_tree` | Cut | `OBJ_EVENT_GFX_CUT_TREE` |
| `smash_rock` | Rock Smash | `OBJ_EVENT_GFX_BREAKABLE_ROCK` |
| `strength_boulder` | Strength (pushable) | `OBJ_EVENT_GFX_PUSHABLE_BOULDER` |

Each is emitted as an overlay record (location + type) so the agent knows *which HM unblocks it*:

```json
"obstacles": [
  { "tile": [9, 4], "type": "cut_tree" },
  { "tile": [11, 6], "type": "strength_boulder" }
]
```

## Perceived (categorical heads)

| Field | Values | RAM source |
|-------|--------|------------|
| `map_id` | FireRed map id | `gMapHeader` map group/number |
| `location_name` | "ROUTE 4", "VIRIDIAN CITY", … | `regionMapSectionId` |
| `context` | `outdoor` / `indoor` / `cave` | map type (`MAP_TYPE_*`) |
| `weather` | weather id (mostly `none` in FRLG) | map header weather |

NPCs and item balls also come from the object-event table at runtime (positions + graphics id); keep
them in the object overlay alongside destructibles, separate from the static grid.

---

## Derived layer

Computed from `perceived.map_id` against `oggen3_data/` and the static map graph — **never trained
into the encoder**:

- map connections (which routes/cities are adjacent) — map-header connection data
- shortest path to a target city — search over the map graph
- avg wild / trainer levels, dominant types per route — encounter/trainer tables

Map lives in the weights; the stat sheet rides alongside in JSON and can be retrained for free.

---

## Caveats

1. **Elevation / bridges break locality.** Some tiles' passability depends on the player's current
   elevation (over/under bridges, multi-level stairs); identical pixels, different collision. Feed the
   player's elevation as a side input, or accept it as residual error. Small fraction of tiles.
2. **Objects are dynamic.** NPCs, cut trees, smashable rocks, and boulders move or disappear and are
   *not* in the static collision layer. The encoder predicts static geometry; the object overlay is
   OR'd in at runtime from the object-event table. Don't make the encoder track moving sprites.
3. **Edge tiles** are partially scrolled — define the 15×10 grid relative to the player-centred camera.

---

## Eval

Because memorisation is the goal, **do not hold out whole routes** — train on the entire world.
Instead hold out *states within* areas (different party, time-of-day tint, menu overlays, NPC
positions) to confirm the encoder keys on the **map geometry**, not on the HUD or party sprite.
Primary metric: per-class tile accuracy / IoU on the walkability grid, with special attention to the
rare-but-critical classes (ledges, warps, destructibles, tall grass).
